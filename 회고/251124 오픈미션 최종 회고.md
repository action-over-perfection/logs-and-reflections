블로그 배포했어요!

18일 이후에는 [중간 회고](https://github.com/action-over-perfection/logs-and-reflections/blob/main/회고/251118%20중간%20회고.md)에서 얘기한대로 에러 핸들링과 api response 구조를 다듬는 등 CRUD 기능을 보강하는데 힘을 쓴 것 같습니다.

우테코 프리코스 3주차 미션에서도 TDD 방식을 도입하다보니 설계에 시간을 많이 쏟았었는데, 이번에도 비슷하게 흘러갔던 것 같아요.

에러 핸들링과 api response 구성 방식이 정말 다양하고, 공식문서에도 명확한 설명이 없었던지라 다양한 방식을 알아보고 제 상황에 맞는 방법을 찾느라 시간을 많이 투자했습니다. 특히 백엔드 입문자로서 알고 있는 **키워드**가 적었다는 점에서 한계를 가장 많이 느꼈던 것 같습니다. *(어떤 것을 모르는지 모르는.. 그런 느낌입니다!)*<br> 
제가 이 프로젝트에서 일반 블로그보다 공식문서를 지향하는 것을 제약사항으로 정했지만, 견문을 넓히는데에는 많은 사람들의 경험과 이야기가 훨씬 효과적인 것 같았습니다. 사실 이 과정에서 키워드를 얻기 위해 ai를 쓰기도 했습니다.

이번 작업하면서 가장 새로웠던 지식은 바로 **Lombok** 이라는 라이브러리였던 것 같습니다.

base response 구조를 잡으면서 데이터를 같이 보내야하는 경우도 있지만 (get api 등), 데이터를 보내지 않아도 되는 경우도 있었는지라 (delete api 등)<br>
후자일 때 제네릭 타입인 데이터를 어떤 형태로 보내야할지 고민이 많았습니다. 물론 `null`을 쓰면 간단하게 해결될 문제였겠지만, 이전에 코틀린을 다루면서 null 사용을 최대한 지양했다보니 습관처럼 이번 작업때도 더 나은 방법이 없는지 생각해보게 되더라고요.

그러다가 탐색 과정에서 builder 패턴에 대해 알게 되었습니다. 전반적으로 생성자와 기능이 흡사하지만, 특정 멤버 변수로만 객체를 생성할 수 있다는 차이점이 매력적으로 다가왔습니다.
그리고 이 builder 패턴은 Lombok 라이브러리를 활용함으로써 보일러플레이트 코드를 크게 줄일 수 있었습니다.<br>
생성자와 getter 등도 어노테이션으로 해결이 가능했던 편리함을 느끼기도 했기 때문에, 오픈미션 이후에 롬복을 깊게 공부하고 싶습니다.

### 배포에 대하여
팀원들과 함께 안드로이드 앱을 한번 배포했던 적이 있는데, 그게 배포 경험의 전부였던지라 이번 프로젝트에서 프론트엔드, 백엔드 둘다 꼭 배포를 해보고 싶었습니다.

비용이 부담되지 않는 선에서 실무와 가까운 배포 경험을 해보고 싶었기 때문에 서버는 무조건 AWS로 진행하겠다는 목표를 가지고 있었습니다. 그리고 이번 프로젝트에선 백엔드에 집중하고자 프론트엔드는 v0을 통해 작업하고, vercel로 간단하게 배포했던 것 같습니다.

원래는 EC2만 사용해서 배포하려고 했지만, Docker 환경 내에서 MySQL 데이터베이스를 관리하고 있었기 때문에 프리티어 서버로는 Spring Boot 프로젝트와 함께 작동시키는 것을 무거워하는 것 같았습니다. 애플리케이션 연결부터 무한 로딩이 발생해서 배포가 불가능했기 때문에 이 기회에 데이터베이스는 RDS로 관리하는 방향으로 작업해보았습니다.<br>
RDS는 처음에는 어렵다고 생각했던 것 같은데 막상 적용해보니 EC2 인스턴스 생성 과정과 비슷한 난이도였던 것 같습니다.<br>
그리고 RDS로 db를 분리해준 후 드디어 서버 배포에 성공했기 때문에... 이 과정에서도 도전과 실행의 중요성을 느끼게 되었습니다.

어떤 분이 제게 "생각의 반대는 실행" 이라고 얘기해주신 적이 있습니다.<br>
전 여전히 생각이 많은 사람이고, 여전히 고민하느라 제자리에 오래 서있기도 하지만, 이번에 저만의 블로그를 만들어보면서 이 많은 생각들을 실행으로 옮기는 경험을 많이 해볼 수 있었던 것 같습니다.<br>
기본 중 기본인 간단한 CRUD API를 만드는게 거의 전부이긴 했지만, Spring 프로젝트를 생성해본 적도 없던 3주 전 저와 비교를 해보면 꽤 많이 성장한 것 같습니다.<br>
우테코 프리코스 과정을 시작하기 전 제 모습과 대조해봐도 개발 실력 측면에서 훨씬 더 나은 사람이 된 것 같기도 합니다.

아키텍처와 설계를 두려워했던지라 (어쩌면 도피의 의도로) 기능 구현만 많이 했던 모습을 고치고 더 나은 개발자가 되기 위해 백엔드에 입문하기로 했고, 우테코에 지원했었습니다.<br>
우테코 프리코스 과정을 통해 제가 제대로 몰입하고, 일도 맡고싶은 또다른 분야를 찾은 것 같습니다.

오픈미션이 종료되고 나서도 이 블로그를 꾸준히 개선해나가고자 합니다.<br>
